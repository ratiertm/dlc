Create `skill/references/test-stage.md` following the exact stage adapter pattern from do-stage.md. Target ~300 lines. Structure:

**## When This Runs** -- user says "test", "verify", "check"; state.json current.stage = 3

**## Prerequisites** -- .lifecycle/ initialized, state.json + manifest.json exist, DO stage must have outputs (gate 2_to_3)

**## Step 0: Stage Initialization** -- Mirror do-stage.md Step 0:
1. Read state.json, verify current.feature and current.stage = 3
2. Check gate 2_to_3: `artifacts.2_do.outputs.length > 0`
3. Update state.json: stage=3, stage_name="TEST", status="in_progress"
4. Read spec from `.lifecycle/features/{feature-name}/spec.md` -- verify `status: implementing`, at least one step `implemented`
5. Read prototype from `.lifecycle/features/{feature-name}/prototype.html` -- parse manifest JSON from `<script type="application/json" id="prototype-manifest">`
6. Read Deviations section from spec.md -- build deviation lookup (DEV-NNN -> spec step mapping with approved status)

**## Step 1: Spec Step Enumeration** -- Parse all spec steps (`## e2e-{feature}-{NNN}`), extract ID, title, chain type, status, verification criteria. Display verification plan:
```
TEST Stage -- {feature-name}
===========================
Spec: .lifecycle/features/{name}/spec.md
Steps to verify: N
Approved deviations: M
```

**## Step 2: Per-Step Spec Verification** (addresses SPEC-04) -- For each step with `implemented` status:
a. Read step's Verification Criteria from spec.md
b. Check deviations -- if step has an approved deviation (DEV-NNN), adjust expected behavior to match deviation's "Actual" field
c. Search codebase for `// SPEC: e2e-{feature}-{NNN}` comment -- record found/not-found and file:line
d. At the SPEC comment location, verify implementation matches verification criteria by reading surrounding code
e. Set step status: `verified` if ALL criteria pass, `failed` if ANY criterion fails
f. Update step status in spec.md on disk: `**Status:** implemented` -> `**Status:** verified` or `**Status:** failed`
g. Re-display progress after each step

Critical: MUST read spec.md from disk (not memory). MUST account for approved deviations. Partial implementation = fail (no "partial" status).

**## Step 3: Prototype Structural Comparison** (addresses PROTO-06) -- STRUCTURAL only, not visual:
a. Parse prototype manifest JSON (specMapping, screens, interactions, fields, errorStates)
b. For each screen in manifest: check that corresponding route/page exists in implementation
c. For each specMapping entry: check that the mapped spec step is implemented (cross-reference with Step 2 results)
d. For each interaction: check that the action handler exists in code
e. For each field: check that the input/form field exists in code
f. For each errorState: check that the error condition is handled in code
g. Record: expected count, found count, missing items per category

Critical: Do NOT compare visual appearance, CSS, colors, or layout. Prototype is for flow validation, not design.

**## Step 4: User Behavioral Verification Checklist** -- Generate concrete user-performable actions from spec steps:
- Screen step -> "Open {route} in browser. Verify {elements} are visible."
- Connection step -> "Perform {action}. Verify {request indicator}."
- Processing step -> "Verify {feedback} appears."
- Response step -> "Verify {result} is displayed."
- Error step -> "Test {condition}. Verify {error behavior}."

Number sequentially. Include spec step ID reference per checklist item.

**## Step 5: Completion** -- Generate outputs and update state:
a. Generate `verification.json` at `.lifecycle/features/{feature-name}/verification.json` with format from 06-RESEARCH.md (specVerification, prototypeVerification, userVerification sections)
b. Generate `verification.md` at `.lifecycle/features/{feature-name}/verification.md` with format from 06-RESEARCH.md (tables for spec results, prototype comparison, SPEC comment traceability, user checklist)
c. Update spec.md YAML frontmatter: if all steps verified -> `status: verified`; if any failed -> `status: failed`
d. Register outputs in manifest.json under `artifacts.3_test.outputs` (type: "verification" for both files, type: "test-report" for verification.md)
e. Determine overall result:
   - If any spec step FAILED or prototype has missing items -> overall FAILED
   - If all structural checks pass -> announce "Claude structural verification PASSED. User behavioral verification PENDING."
   - Present the behavioral checklist to user
   - COMMIT gate blocked until user confirms behavioral verification
f. On FAIL: report failures to user, let user decide direction (DO NOT auto-fix, DO NOT auto-transition back to DO). Per CONTEXT.md locked decision.
g. On full PASS (structural + user behavioral confirmed): update state.json status="completed", set resume_hint

**## Anti-Patterns** -- Table with 6 rows:
1. Auto-fixing failures (Claude must NOT fix, user decides)
2. Skipping user behavioral verification (even if structural passes)
3. Working from memory instead of reading spec from disk
4. Combining structural and behavioral into one pass (track independently)
5. Comparing prototype visual appearance (structural only, not CSS/colors)
6. Ambiguous pass/fail (no "partial" -- each step is verified or failed)

**## Relationship to Other Skills:**
- dev-lifecycle: Primary skill for Stage 3 (spec + prototype verification)
- GSD: Available as supporting tool (`/gsd:verify-work`) for broader verification
- PDCA: Available as supporting tool (`/pdca analyze`) for gap analysis
- Note the role shift: dev-lifecycle is now primary for Stage 3 (was GSD in original SKILL.md)

**## File Locations** -- Table listing: this adapter, spec format reference, prototype format reference, stage transitions, verification output paths, state template, manifest template
