- HEAD section: `<meta charset="UTF-8">`, `<meta name="viewport">`, title with `{Feature Name}` placeholder, inline CSS (~30 lines) with system-ui font stack, responsive max-width (480px mobile, 640px desktop via `@media (min-width: 768px)`), wireframe styling (neutral colors, no gradients/shadows/brand colors), `.screen` class hidden by default with `.active` display:block, form input styling, button styling with hover, `.error` and `.success` message styling, `.field-group` label styling, `.nav-link` styling
- MANIFEST section: `<script type="application/json" id="prototype-manifest">` with the canonical manifest schema from 03-RESEARCH.md. All fields present with `{placeholder}` values: feature, specFile, version, generatedFrom, screens[], specMapping[], interactions[], fields[], errorStates[]
- SCREENS section: `<div id="app">` containing two example `<div data-screen="{screen-name}" data-spec-id="e2e-{feature}-001" class="screen">` divs with placeholder content showing data-field, data-action, data-error attribute examples
- ENGINE section: `<script>` with hash router (showScreen function using classList add/remove 'active'), getDefaultScreen from manifest, hashchange listener, delegated click handler for `[data-action]` elements, handleAction stub function, initialization call on load

Include HTML comments as section delimiters: `<!-- SECTION N: NAME -->`.

**2. `skill/references/prototype.md`** -- Format reference document (100+ lines) covering:

- Purpose section: Why prototypes exist (flow validation, not design), relationship to E2E spec
- Four-Section Layout: HEAD / MANIFEST / SCREENS / ENGINE with what belongs in each
- Data Attribute Convention Table: all 8 attributes from 03-RESEARCH.md (data-screen, data-action, data-field, data-error, data-spec-id, data-nav, data-state, data-mock) with Purpose, Placement, Example columns
- Manifest Schema: Document each field (feature, specFile, version, generatedFrom, screens, specMapping, interactions, fields, errorStates) with types and descriptions
- Spec Linking Rules: Dual linking requirement (data-spec-id attribute + manifest specMapping entry), one data-spec-id per spec step (children inherit via DOM hierarchy), granularity rules (Screen step -> container div, Connection step -> clickable element, Error step -> error display element)
- Hash Routing Rules: hashchange listener, default screen on load, screen-as-div pattern
- Anti-Patterns: over-styling, external dependencies, History API routing, alert-based interactions, separate manifest file, data-spec-id on wrong granularity
- File Locations: `.lifecycle/features/{name}/prototype.html` for project output, `skill/templates/prototype-template.html` for template, `skill/examples/` for examples

Do NOT include: CSS framework recommendations, v2 features, or anything deferred.
