Create `skill/references/commit-stage.md` following the exact same document structure as test-stage.md (the canonical Stage adapter pattern). The file must contain these sections in order:

**Header:** "# COMMIT Stage Adapter" with one-line description matching test-stage.md style.

**When This Runs:**
- User says "commit", "save", or similar
- state.json current.stage = 4 (or transitioning from 3 to 4)
- SKILL.md Stage 4 section directs here via Read directive

**Prerequisites:**
- .lifecycle/ directory initialized
- state.json and manifest.json exist
- TEST stage must have outputs -- gate 3_to_4 must pass

**Step 0: Stage Initialization**
1. Read state.json, verify current.feature and current.stage = 4 (or transition from 3)
2. Check basic gate: artifacts.3_test.outputs.length > 0
3. If gate fails: set status "blocked", display missing artifacts, STOP
4. Content-level gate: read verification.json from .lifecycle/features/{feature-name}/verification.json
   - Check specVerification.overall === "passed"
   - Check userVerification.status === "passed"
   - If specVerification.overall === "failed": display gap list (see Step 1 blocked format), STOP
   - If userVerification.status === "pending": display "User behavioral verification not completed", STOP
   - If userVerification.status === "failed": display gap list with user-reported failures, STOP
5. Verify required artifacts exist on disk: spec.md, prototype.html, verification.json, verification.md
6. Update state.json: stage=4, stage_name="COMMIT", status="in_progress", progress.current_stage_started_at

**Step 1: Verification Gate Display**
When blocked (any check in Step 0 fails), display this format:
```
COMMIT Stage -- {feature-name} -- BLOCKED
==========================================
Verification gate FAILED. Cannot commit.

Failed spec steps:
  [FAIL] e2e-{feature}-{NNN}: {title} -- {reason}
  [FAIL] e2e-{feature}-{NNN}: {title} -- {reason}

User behavioral verification: {PASSED/FAILED/PENDING}

Options:
  1. Return to DO stage to fix failures
  2. Return to TEST stage to re-verify
  3. Override and commit anyway (not recommended)
```

If user chooses override (option 3):
- Display explicit warning: "WARNING: Committing with verification failures. This will be recorded."
- Require user to type "override" to confirm
- Continue to Step 2 with override flag set to true
- Record override in commit message and manifest output

**Step 2: Commit Preparation**
1. Collect changed files from manifest.json 2_do outputs (type "code" entries)
2. Detect ADR references from manifest.json 2_do outputs where type="adr"
3. Read spec.md for: feature name, step count, status
4. Read deviations count from spec.md Deviations section
5. Generate why-centric commit message using this format:
```
{type}({feature-name}): {why-summary}

{1-2 sentence description of the user/business value delivered}

Spec: .lifecycle/features/{name}/spec.md
Steps: {N} verified ({M} deviations)
{If ADRs: (ADR-NNN) {adr title} -- one line per ADR}
{If override: WARNING: Committed with verification override}
```
6. Present commit message to user for approval or editing
7. Wait for user confirmation before proceeding

**Step 3: Git Commit Execution**
1. Stage files:
   - All source code changes (from DO stage)
   - .lifecycle/features/{name}/spec.md
   - .lifecycle/features/{name}/prototype.html
   - .lifecycle/features/{name}/verification.json
   - .lifecycle/features/{name}/verification.md
   - docs/decisions/*.md (any ADRs created during DO)
2. Exclude from staging:
   - .env, credentials, secrets
   - node_modules/, .venv/, etc.
   - .lifecycle/state.json (runtime state)
   - .lifecycle/manifest.json (runtime state)
   - .lifecycle/history/ (runtime state)
3. Execute git commit with approved message
4. Capture commit hash
5. Optionally create version tag if user requests (format: v{major}.{minor}.{patch})

**Step 4: Completion**
1. Register outputs in manifest.json under artifacts.4_commit.outputs:
   - { type: "commit-hash", path: "{git-sha}", created_at: ISO-8601, status: "complete" }
   - If tag created: { type: "tag", path: "{tag-name}", created_at: ISO-8601, status: "complete" }
   - If override: add "override": true to commit-hash entry
2. Set artifacts.4_commit.completed_at
3. Update state.json: status="completed", resume_hint="COMMIT complete. Ready for DEPLOY stage."
4. Write history entry to .lifecycle/history/ (per stage-transitions.md format)
5. Announce completion:
```
COMMIT Stage -- {feature-name} -- COMPLETE
==========================================
Commit: {short-sha} {first line of message}
{If tag: Tag: {tag-name}}
{If override: NOTE: Committed with verification override}

Inner loop complete. Ready for outer loop (DEPLOY) or next feature.
```

**Anti-Patterns section** (table format matching test-stage.md):
1. Auto-committing without user confirmation -- must present message and get approval
2. Skipping verification.json content check -- basic outputs.length > 0 is insufficient
3. Including sensitive files -- must exclude .env, credentials, .lifecycle/ runtime state
4. Committing when userVerification is "pending" -- dual gate requires BOTH verifications
5. What-centric commit messages -- focus on WHY (business value), not WHAT (file changes)
6. Forgetting to register commit outputs in manifest -- breaks gate 4_to_5 for DEPLOY

**Relationship to Other Skills section:**
- dev-lifecycle: Orchestrates COMMIT Stage workflow, enforces verification gate
- Git: Direct tool for commit execution (per role-matrix)
- ADR: Referenced in commit messages when ADRs exist from DO stage
- GSD, PDCA: Not directly involved in Stage 4
