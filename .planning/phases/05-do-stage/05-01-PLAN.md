  <action>
Create `skill/references/do-stage.md` following plan-stage.md's exact structural pattern. The file MUST contain these sections in order:

1. **Header** -- "DO Stage Adapter" title, one-line purpose ("This reference defines the complete DO Stage workflow -- Stage 2 of the dev-lifecycle pipeline.")

2. **When This Runs** -- Stage 2 entry conditions (user says "implement", "build", "do", state.json stage = 2, SKILL.md Stage 2 directs here)

3. **Prerequisites** -- Approved spec required (gate 1_to_2 must pass), `.lifecycle/` initialized

4. **Step 0: Stage Initialization** -- Read state.json, check gate 1_to_2 (`artifacts.1_plan.outputs.length > 0 && artifacts.1_plan.completed_at != null`), update state.json (stage=2, stage_name="DO", status="in_progress"), read approved spec from `.lifecycle/features/{feature-name}/spec.md`, verify frontmatter `status: agreed`, update spec status to `status: implementing`

5. **Step 1: Spec Load and Checklist Build** -- MUST read spec.md from disk (not memory), parse all spec steps (## e2e-{feature}-{NNN} headings), extract Status fields, build and display checklist format:
```
DO Stage -- {feature-name}
======================
Spec: .lifecycle/features/{feature-name}/spec.md
Status: implementing (N/M steps done)

Checklist:
  [x] e2e-{feature}-001: {title} ({chain})
  [ ] e2e-{feature}-002: {title} ({chain})
...
Deviations: 0 recorded
ADRs created: 0
Next step: e2e-{feature}-{NNN} ({chain})
```

6. **Step 2: Per-Step Implementation Loop** -- For each spec step with `**Status:** pending`:
   a. Read the step's details (Chain type, What section, Details section)
   b. Implement the step in code
   c. Add `// SPEC: e2e-{feature}-{NNN} -- {step title}` comment at the implementation site
   d. Update the step's `**Status:**` from `pending` to `implemented` in spec.md
   e. Re-display the checklist with updated progress
   f. If implementation must differ from spec -> go to Step 3 (Deviation Handling)

7. **Step 3: Deviation Handling** -- When implementation must differ from spec:
   a. Pause implementation immediately
   b. Present deviation to user in this exact format:
   ```
   Spec Deviation Detected
   ============================
   Step: e2e-{feature}-{NNN} ({step title})
   Spec says: {original spec detail}
   Implementing as: {proposed change}
   Reason: {why the change is needed}
   Continue? (yes -> record deviation, continue / no -> implement per original spec)
   ```
   c. If user approves: append to `## Deviations` section in spec.md using format from spec-template.md (DEV-NNN, 5 fields: Spec step, Original, Actual, Reason, Approved: yes)
   d. If user rejects: implement per original spec, no deviation recorded
   e. DEV numbering: sequential within the feature (DEV-001, DEV-002, etc.)
   f. Return to Step 2 loop

8. **Step 4: ADR Detection** -- Two active triggers per CONTEXT.md:
   a. **Deviation trigger:** When a deviation is recorded AND it involves choosing a different technology/pattern/approach (not minor field/path changes) -> suggest ADR
   b. **Tradeoff trigger:** When conversation contains tradeoff language:
      - "A instead of B" / "A 대신 B"
      - "chose X over Y" / "X로 결정"
      - "tradeoff" / "trade-off" / "트레이드오프"
      - "intentional tech debt" / "나중에 바꿔야"
      - Comparing 2+ options with pros/cons
   c. Suggestion format: "이 결정은 ADR로 기록할 가치가 있습니다. ADR을 생성할까요?"
   d. If user agrees: delegate to ADR skill's 6-step workflow (do not hand-roll ADR format)
   e. After ADR created: insert WHY+SEE comments in code. When a step also has a SPEC comment, order is: `// SPEC:` first, then `// WHY:`, then `// SEE:`
   f. ADR threshold: decision must involve (a) choosing between 2+ viable approaches AND (b) having lasting consequences. Routine implementation choices don't qualify.

9. **Step 5: Completion** -- When all spec steps have `**Status:** implemented`:
   a. Register outputs in manifest.json under `artifacts.2_do.outputs`:
      - type: "code", path: changed files list, status: "complete"
      - type: "spec-updated", path: spec.md path, status: "complete"
      - Any ADR documents created
   b. Set `artifacts.2_do.completed_at` to current ISO 8601 timestamp
   c. Update state.json: `current.status = "completed"`, `session.resume_hint = "DO complete. Ready for TEST stage."`
   d. Display final checklist showing all steps implemented, deviation count, ADR count
   e. Announce: "DO complete. Ready for TEST stage."

10. **Anti-Patterns** -- Table with at least these 6 entries:
    | # | Anti-Pattern | Why It Is Wrong |
    - Starting DO without approved spec (gate exists for a reason)
    - Silently deviating from spec (defeats traceability)
    - Auto-approving deviations (user confirmation is mandatory)
    - Skipping SPEC comments (breaks traceability thread)
    - Creating ADRs for every deviation (causes suggestion fatigue)
    - Working from memory instead of reading spec from disk (IDs may not match)

11. **Relationship to Other Skills** -- dev-lifecycle is primary. PDCA available as supporting (user can invoke directly). ADR delegates to existing ADR skill for document creation. GSD not involved in Stage 2.

12. **File Locations** -- Table listing: this adapter, spec format reference, spec template, stage transitions, state template, manifest template. Use `$CLAUDE_SKILL_DIR` variable. Include both runtime and version-controlled paths.

Target length: 250-350 lines (similar to plan-stage.md's 311 lines). Do NOT pad with unnecessary content but do NOT omit required details either.
