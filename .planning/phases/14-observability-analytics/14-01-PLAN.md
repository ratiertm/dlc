```
# Observability & Analytics

This reference defines all observability features: session context tracking, stage transition analytics, rework event detection, spec baseline diffing, and time-per-stage metrics.

Data files live in `.lifecycle/sessions/` and `.lifecycle/analytics/`. All analytics are append-only and non-blocking (analytics write failure never blocks stage transitions).
```

**Section 1: Session Context (OBSV-01)**

Title: `## Session Context Files`

Define the complete session context lifecycle:

1. **Creation (at Session Start step 6):**
   - Directory: `.lifecycle/sessions/` (create if not exists)
   - Filename: `{ISO-date}T{HH-MM-SS}Z-{feature}.md` (e.g., `2026-03-25T14-00-00Z-user-auth.md`)
   - Initial content template:
     ```markdown
     # Session: {ISO-timestamp}

     **Feature:** {feature-name}
     **Stage at start:** {N} {NAME}
     **Mode:** {mode}
     **Started:** {ISO-timestamp}

     ## Decisions Made
     (none yet)

     ## Artifacts Touched
     (none yet)

     ## Stage Transitions
     (none yet)

     ## Notes
     {resume_hint from state.json, or "New session"}
     ```

2. **Update (at stage transitions and session end):**
   - Append new decisions (delta from `.lifecycle/decisions.md` -- compare line count at session start vs now)
   - Append artifacts touched (new entries in manifest.json since session start)
   - Append stage transitions that occurred
   - Update "Stage at end" field
   - Add computed duration

3. **Constraints:**
   - Cap session file at ~50 lines. If exceeding, summarize older entries.
   - Only record deltas (new items since session start), not full state.

**Section 2: Stage Transition JSONL (OBSV-02)**

Title: `## Stage Transition Analytics`

Define JSONL append logic that runs alongside existing history writes:

1. **When:** After Transition Procedure Step 5 (write history entry), as Step 5b
2. **Directory:** `.lifecycle/analytics/` (create if not exists)
3. **File:** `.lifecycle/analytics/stage-transitions.jsonl`
4. **Format (one JSON line per transition):**
   ```json
   {"timestamp":"ISO-8601","feature":"{name}","from_stage":{N},"from_name":"{NAME}","to_stage":{N+1},"to_name":"{NAME}","direction":"forward","mode":"{mode}","gate_result":"{passed|failed|skipped}","duration_in_stage_seconds":{seconds}}
   ```
5. **`duration_in_stage_seconds` computation:** Current timestamp minus `progress.current_stage_started_at` from state.json, converted to integer seconds.
6. **`direction` values:** "forward" (N to N+1), "backward" (N to M where M < N), "skip" (auto-skipped stages).
7. **Non-blocking:** If JSONL write fails, log warning and continue transition normally.

**Section 3: Rework Events (OBSV-03)**

Title: `## Rework Event Tracking`

Define backward transition detection and logging:

1. **When:** On any backward stage transition (to_stage < from_stage)
2. **File:** `.lifecycle/analytics/rework-events.jsonl`
3. **Format:**
   ```json
   {"timestamp":"ISO-8601","feature":"{name}","from_stage":{N},"from_name":"{NAME}","to_stage":{M},"to_name":"{NAME}","reason":"{why backward}","impact":"{what needs redo}","rework_count":{running_total}}
   ```
4. **`rework_count`:** Count existing entries in rework-events.jsonl for this feature, add 1.
5. **`reason`:** From the backward transition trigger (e.g., "3 spec steps failed verification", "user requested re-plan").
6. **`impact`:** What work is affected (e.g., "re-implement steps 002, 004").
7. **Also write to stage-transitions.jsonl** with `direction: "backward"` (dual-write: both files get an entry).

**Section 4: Spec Baseline Snapshot (OBSV-04)**

Title: `## Spec Baseline & Diff`

Define baseline creation and on-demand diff:

1. **Baseline creation (at PLAN completion, before DO transition):**
   - Directory: `.lifecycle/analytics/spec-baselines/` (create if not exists)
   - File: `.lifecycle/analytics/spec-baselines/{feature}.baseline.md`
   - Action: Copy `.lifecycle/features/{feature}/spec.md` to baseline path
   - Guard: Only create if baseline does NOT already exist for this feature (preserve original)
   - If user explicitly wants a reset: use `{feature}.baseline.v{N}.md` versioned naming

2. **On-demand diff (when user asks "spec diff" or "what changed"):**
   - Read baseline: `.lifecycle/analytics/spec-baselines/{feature}.baseline.md`
   - Read current: `.lifecycle/features/{feature}/spec.md`
   - Compare section by section:
     - Status changes (pending -> implemented -> verified)
     - Deviations added
     - Steps added or removed
   - Present structured diff report:
     ```markdown
     # Spec Diff: {feature}

     **Baseline:** {date}
     **Current:** {date}

     ## Status Changes
     | Step | Baseline | Current |
     |------|----------|---------|
     | e2e-{feature}-001 | pending | verified (attempt 1) |

     ## Deviations from Baseline
     - DEV-001: {description} (approved)

     ## Summary
     - Steps: {total} total, {verified} verified, {implemented} implemented
     - Deviations: {count}
     - Completeness delta: baseline N/A -> current {score}/10
     ```

**Section 5: Time-Per-Stage Metrics (OBSV-05)**

Title: `## Time-Per-Stage Metrics`

Define time computation and on-demand reporting:

1. **Data source:** `stage-transitions.jsonl` `duration_in_stage_seconds` field (computed at each transition)
2. **On-demand report (when user asks "time per stage" or "metrics" or "analytics"):**
   - Read `.lifecycle/analytics/stage-transitions.jsonl`
   - Filter by feature (or show all features if requested)
   - Compute per-stage durations
   - Identify bottleneck (stage with longest duration)
   - Present report:
     ```markdown
     # Time-Per-Stage: {feature}

     | Stage | Name | Duration | % of Total |
     |-------|------|----------|------------|
     | 1 | PLAN | 1h 0m | 25% |
     | 2 | DO | 2h 30m | 62% |
     | 3 | TEST | 30m | 13% |

     **Bottleneck:** Stage 2 (DO) -- 2h 30m (62% of total)
     **Total cycle time:** 4h 0m
     **Rework overhead:** 30m (1 rework event)
     ```
3. **Rework overhead:** Sum durations of stages entered via backward transitions (direction: "backward" in JSONL).

**Section 6: On-Demand Report Triggers**

Title: `## Analytics Commands`

Define the trigger keywords and corresponding actions:

| User Says | Action |
|-----------|--------|
| "analytics" / "metrics" / "time per stage" | Generate time-per-stage report for current feature |
| "spec diff" / "what changed from spec" | Generate spec baseline diff for current feature |
| "session history" / "sessions" | List session files, summarize recent sessions |
| "rework report" / "rework events" | Summarize rework-events.jsonl for current feature |
| "stage transitions" / "transition history" | Summarize stage-transitions.jsonl for current feature |

**Section 7: Directory Setup**

Title: `## Directory Structure`

```
.lifecycle/
  sessions/                          # Session context files (OBSV-01)
    {ISO-date}T{HH-MM-SS}Z-{feature}.md
  analytics/                         # Structured analytics (OBSV-02/03/05)
    stage-transitions.jsonl
    rework-events.jsonl
    spec-baselines/                  # Spec snapshots (OBSV-04)
      {feature}.baseline.md
```

Note: directories are created on first write (not eagerly at project init).

**Section 8: Error Tolerance**

Title: `## Error Handling`

- JSONL reads: skip malformed lines (each line is independent)
- Missing analytics directory: create on first write
- Missing session file: create fresh (mid-session recovery)
- Analytics write failure: warn user, continue normal operation (non-blocking)
- Empty JSONL file: report "No data yet" (not an error)
