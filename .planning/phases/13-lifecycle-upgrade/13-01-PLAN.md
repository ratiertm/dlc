Create `skill/references/lifecycle-upgrade.md` containing the complete migration procedure Claude follows when a version mismatch is detected. Structure:

**Section 1: When Triggered**
- Session Start detects skill/VERSION > .lifecycle/.dlc-version (or .dlc-version missing)
- Only runs when .lifecycle/ directory exists (fresh installs skip migration)

**Section 2: Pre-Migration Backup (UPGR-03)**
- Create .lifecycle/.upgrade-backup/ directory
- Copy these files (if they exist) into backup: state.json, manifest.json, config.yaml, settings-changelog.md, decisions.md, LIVING-STATE.md
- If backup creation fails: abort migration, display error, continue session with old schema

**Section 3: Migration Registry**
Document the ordered, idempotent migration blocks:

```
## Migration Registry

### v1.0 -> v2.0

Marker: `.lifecycle/.v2-migrated`

Pre-check: If `.lifecycle/.v2-migrated` exists, SKIP this entire block.

Steps (each is idempotent -- safe to re-run):

1. **state.json schema update:**
   - Read .lifecycle/state.json
   - If `current.completeness` field missing: add `"completeness": null` to `current` object
   - If `version` field is "1.0": change to "2.0"
   - Preserve ALL existing fields (stage, status, feature, mode, progress, session)
   - Write updated state.json

2. **config.yaml creation:**
   - If .lifecycle/config.yaml does NOT exist: copy from $CLAUDE_SKILL_DIR/templates/config.yaml
   - If exists: leave unchanged

3. **settings-changelog.md creation:**
   - If .lifecycle/settings-changelog.md does NOT exist: copy from $CLAUDE_SKILL_DIR/templates/settings-changelog.md
   - If exists: leave unchanged

4. **decisions.md creation:**
   - If .lifecycle/decisions.md does NOT exist: copy from $CLAUDE_SKILL_DIR/templates/decisions.md
   - If exists: leave unchanged

5. **Version tracking:**
   - Write "2.0.0" to .lifecycle/.dlc-version
   - Touch .lifecycle/.v2-migrated (empty marker file)
```

**Section 4: Rollback Procedure (UPGR-03)**
- If ANY step in the migration block fails:
  1. Restore all files from .lifecycle/.upgrade-backup/ (overwrite current versions)
  2. Delete .lifecycle/.upgrade-backup/
  3. Do NOT create .v2-migrated marker
  4. Display: "Migration to v2.0 failed. Rolled back to previous state. Error: {description}"
  5. Continue session with old schema (best effort)

**Section 5: Post-Migration Cleanup**
- On success: delete .lifecycle/.upgrade-backup/ directory
- Display changelog summary (see Section 6)

**Section 6: Changelog Display (UPGR-04)**
```
dev-lifecycle upgraded: v{old} -> v{new}

What's new:
- [Read skill/CHANGELOG.md, extract bullets between old and new version]

Continuing session...
```

**Section 7: Future Migration Blocks**
- New versions add blocks AFTER v2.0 block (e.g., "### v2.0 -> v3.0")
- Each block has its own marker file
- Blocks run sequentially: a user upgrading from v1.0 to v3.0 runs v1.0->v2.0 first, then v2.0->v3.0
- Each block's pre-check ensures it only runs once

**Section 8: Edge Cases**
| Scenario | Behavior |
|----------|----------|
| No .lifecycle/ directory | Skip migration entirely (fresh install) |
| .dlc-version missing | Assume v1.0.0 (pre-upgrade install) |
| .v2-migrated exists but .dlc-version missing | Write .dlc-version = "2.0.0", skip migration steps |
| User mid-stage (e.g., DO in progress) | Migration preserves current.stage, status, feature -- no disruption |
| .upgrade-backup/ already exists from prior failed migration | Remove it first, then proceed with fresh backup |
| state.json missing entirely | Let state-reconcile.md handle it first, then run migration on reconciled file |
