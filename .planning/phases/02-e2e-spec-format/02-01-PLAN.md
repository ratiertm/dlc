Create two files:

**1. `skill/templates/spec-template.md`** -- A fillable template that users/Claude fill in per feature.

YAML frontmatter with these exact fields:
```yaml
---
feature: {kebab-case-name}
title: {human-readable title}
created: {ISO-8601 date}
updated: {ISO-8601 date}
status: draft
depends_on: []
steps: {number}
tags: []
---
```

Status lifecycle: `draft -> agreed -> implementing -> verified` (also `failed` from implementing/verified).

Body structure:
- H1 title: `# E2E Spec: {Title}`
- One-line round-trip summary describing user action -> storage -> user response
- 5 step sections, each with heading `## e2e-{feature}-{NNN}: {Step Title}`
- Each step has: `**Chain:** {Screen|Connection|Processing|Response|Error}`, `**Status:** pending`, `### What`, `### Verification Criteria` (checkbox list), `### Details`

Details sections vary by chain type:
- Screen: Element, User Action, Initial State
- Connection: Method, Endpoint, Request, Auth
- Processing: Steps (numbered), **Storage** (mandatory: `{destination} -- {operation}` format, e.g., "PostgreSQL users table -- READ")
- Response: Success Status, Response Shape, UI Updates (list)
- Error: Table with columns Condition | Behavior | Status. Minimum 3 rows: (1) client validation, (2) server/processing, (3) infrastructure/network.

At bottom, include a `## Deviations` section (initially empty) with template:
```
### DEV-NNN: {Brief description}
- **Spec step:** e2e-{feature}-{NNN}
- **Original:** {what spec said}
- **Actual:** {what was implemented}
- **Reason:** {why}
- **Approved:** {yes/no}
```

**2. `skill/references/e2e-spec.md`** -- Format reference document (progressive disclosure from SKILL.md).

Must cover:
- Purpose: why E2E specs exist (the "done but broken" problem)
- 5-step chain overview table: step name, what it covers, required fields
- Spec ID convention: `e2e-{feature}-{NNN}`, kebab-case feature name, zero-padded sequential number
- Cross-artifact linking: spec.md (`## e2e-{feature}-{NNN}`), prototype.html (`data-spec-id`), code (`// SPEC: e2e-{feature}-{NNN}`), verification.json (keyed by spec ID)
- Status lifecycle diagram: draft -> agreed -> implementing -> verified/failed
- PLAN/DO/TEST usage: PLAN creates spec, DO uses as checklist (updates status per step), TEST verifies per-step pass/fail
- Deviation log format and when to use it
- Granularity guide: one spec per feature, each distinct interaction gets its own 5-step chain, soft limit 25 steps (5 interactions) per feature
- Storage field rules: mandatory in Processing, format is `{destination} -- {operation}` (READ, WRITE, READ+WRITE)
- Anti-patterns: vague assertions, skipping chain links, spec without round-trip, over-granular specs
- Manifest integration: spec.md registers in `.lifecycle/manifest.json` under `1_plan.outputs` with type "spec"
- Template location pointer: `$CLAUDE_SKILL_DIR/templates/spec-template.md`
- Example pointer: `$CLAUDE_SKILL_DIR/examples/user-login.spec.md`
