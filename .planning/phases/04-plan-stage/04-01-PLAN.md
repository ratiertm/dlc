Create `skill/references/plan-stage.md` -- the complete PLAN Stage adapter reference. This file is Claude-instructions-as-code: when SKILL.md says "Read: plan-stage.md", Claude follows these instructions to run the PLAN Stage.

Structure the file with these sections:

**Header:**
- Title: "# PLAN Stage Adapter"
- When This Runs: user starts a new feature (state.json current.stage = 1), user says "plan", "start feature", etc.
- Prerequisites: .lifecycle/ directory initialized, state.json and manifest.json exist

**Step 0: Directory Initialization**
- Ensure `.lifecycle/`, `.lifecycle/features/`, `.lifecycle/features/{feature-name}/`, `.lifecycle/history/` all exist
- If state.json missing, copy from `$CLAUDE_SKILL_DIR/templates/state.json`
- If manifest.json missing, copy from `$CLAUDE_SKILL_DIR/templates/manifest.json`
- Set state.json: current.stage=1, current.stage_name="PLAN", current.status="in_progress", current.feature={feature-name}

**Step 1: Preflight Check (non-blocking)**
- Priority order per CONTEXT.md decision:
  1. Retrospective lessons (HIGHEST): Read `.lifecycle/history/` for retrospective artifacts, extract "Lessons for Next Phase"
  2. Related ADRs: Scan `docs/decisions/` for ADRs related to the feature domain
  3. Unresolved gaps: Check manifest.json for previous features with failed verification
  4. Technical debt: Scan related files for TODO/FIXME/HACK comments
- Output format: section per source, each item prefixed with [LESSON], [ADR], [GAP], [DEBT]
- Edge case: First feature ever -- "No previous lessons found. First feature." and proceed
- CRITICAL: Preflight failure = warning only. Do NOT block spec generation because of warnings.

**Step 2: E2E Spec Generation**
- Read directives: `$CLAUDE_SKILL_DIR/templates/spec-template.md`, `$CLAUDE_SKILL_DIR/references/e2e-spec.md`, `$CLAUDE_SKILL_DIR/examples/user-login.spec.md`
- Approach: draft-then-revise (per CONTEXT.md discretion recommendation)
  1. Ask user to describe the feature in their own words
  2. Generate complete draft spec with all 5 chain steps per interaction
  3. Present draft to user for review
  4. Iterate until user satisfied
- Generation checklist (each item MUST be verified):
  - Feature name in kebab-case for frontmatter
  - One-line round-trip summary
  - Each interaction has exactly 5 chain steps (Screen, Connection, Processing, Response, Error)
  - Every Processing step has mandatory Storage field with format: {destination} -- {operation}
  - Error step has minimum 3 conditions (client validation, server error, network error)
  - Spec IDs follow e2e-{feature}-{NNN} convention, sequential across interactions
  - Verification Criteria are concrete and checkable
- Save to: `.lifecycle/features/{feature-name}/spec.md`
- Set frontmatter status: `draft`
- Scope guidance: Recommend 1-3 interactions (5-15 spec steps) for first features. Soft limit: 25 steps per feature.

**Step 3: Prototype Generation**
- Read directives: `$CLAUDE_SKILL_DIR/templates/prototype-template.html`, `$CLAUDE_SKILL_DIR/references/prototype.md`, `$CLAUDE_SKILL_DIR/examples/user-login.prototype.html`
- MUST read the spec just created in Step 2 (not from memory)
- Generation rules (from RESEARCH.md):
  - For each Screen step: create `<div data-screen="name">` with form fields
  - For each Connection step: create `<button data-action="name">` on the screen
  - For each Processing step: create loading screen with data-state="loading" and 800ms auto-transition
  - For each Response step: create result screen with data-mock bindings
  - For each Error step: create `<div data-error="name">` elements
  - Wire handleAction() in ENGINE section
  - Populate MANIFEST JSON: screens, specMapping, interactions, fields, errorStates
  - Enforce dual spec linking: data-spec-id attributes AND manifest specMapping entries
- CRITICAL: Must work with file:// protocol. No fetch(), no external resources, no CDN links.
- Save to: `.lifecycle/features/{feature-name}/prototype.html`
- Anti-pattern: Never generate prototype before spec. Spec is source of truth for IDs.

**Step 4: Agreement Gate (hard blocking)**
- Protocol (per CONTEXT.md locked decision):
  1. Announce: "Prototype saved to .lifecycle/features/{name}/prototype.html"
  2. Provide open command: `open .lifecycle/features/{name}/prototype.html` (macOS)
  3. Instruct: "Please open this file in your browser and click through all interactions."
  4. Ask: "Does this match what you expect? Any changes needed?"
  5. Explicitly ask: "Did you open prototype.html in your browser? Did you click through all interactions?"
  6. IF user requests changes: clarify changes, regenerate spec and/or prototype (always regenerate prototype from scratch if spec changes), return to announcement
  7. IF user approves (says "OK", "approved", "looks good", "yes", etc.):
     a. Update spec frontmatter: status: draft -> agreed
     b. Register spec.md in manifest.json under 1_plan.outputs: {type:"spec", path:".lifecycle/features/{name}/spec.md", created_at:timestamp, status:"complete"}
     c. Register prototype.html in manifest.json under 1_plan.outputs: {type:"prototype", path:".lifecycle/features/{name}/prototype.html", created_at:timestamp, status:"complete"}
     d. Set manifest 1_plan.completed_at to current timestamp
     e. Update state.json: current.status -> "completed"
     f. Announce: "PLAN complete. Ready for DO stage."
- CRITICAL: MUST NOT set 1_plan.completed_at until user says words equivalent to approval. No auto-completion.

**Section: Anti-Patterns**
Include the 6 anti-patterns from RESEARCH.md verbatim:
1. Auto-completing PLAN without user approval
2. Generating prototype before spec
3. Blocking on preflight failures
4. Calling GSD/PDCA for spec generation (dev-lifecycle is independent per CONTEXT.md)
5. Generating spec without Storage field
6. Skipping error conditions (minimum 3 required)

Also deploy to runtime: copy the file to `~/.claude/skills/dev-lifecycle/references/plan-stage.md`
