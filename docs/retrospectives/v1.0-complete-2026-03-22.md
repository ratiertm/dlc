## What Went Well

- **GSD의 discuss→plan→verify→execute 파이프라인이 극도로 효율적이었다.** 10개 Phase를 하루에 모두 완료할 수 있었던 건 이 자동화된 워크플로 덕분. 각 Phase마다 연구→계획→검증→실행이 일관되게 반복되어 품질을 유지하면서 속도가 나왔다.

- **Inner Loop 어댑터 패턴(Step 0-N + anti-patterns + file locations)이 강력했다.** Phase 4에서 plan-stage.md를 만들고 나니, Phase 5-7, 9는 거의 기계적으로 복제할 수 있었다. 패턴이 확립되면 후속 작업이 빨라지는 것을 실감.

- **사용자와의 초기 논의에서 핵심 문제를 정확히 포착했다.** "완성이라 했는데 100% 안 된다", "세팅 맥락이 사라진다", "다시 가르쳐야 한다" — 이 3가지 문제가 모든 설계 결정의 기준이 됨. E2E Spec + Prototype + Living State가 이 문제들에 직접 대응.

- **Plan checker의 revision loop가 품질을 올렸다.** Phase 8에서 4개 blocker가 발견되어 revision 후 재검증 통과. checker 없이 바로 실행했으면 VALIDATION.md가 존재하지 않는 파일을 참조하는 상태로 배포됐을 것.

## What Went Wrong

- **리서치를 두 번 해야 했다.** 처음 리서치에서 User Interaction Prototype과 E2E Spec을 Phase 6에 분리했는데, 사용자가 "이것도 계획에 넣어" 라고 피드백하여 리서치부터 다시 시작. 핵심 기능을 초기에 별도 Phase로 분리한 것이 실수 — 사용자의 핵심 문제와 직결되는 기능은 처음부터 Inner Loop에 내장해야 했다.

- **메모리 문제 해결을 먼저 설계하려 했다.** 사용자가 "무조건 덤비지 말고 역검토해야지"라고 교정. 솔루션을 먼저 설계하고 끼워맞추는 대신, 먼저 만들고 실사용 후 검증하는 순서가 올바름.

- **Phase 9 Outer Loop가 5개 Stage를 한 Phase에 묶었다.** 각 어댑터가 200-250줄로 적지 않은데 2개 task에 5개 파일을 담았다. task 당 2-3개 파일이 적절했을 것. 다행히 패턴이 확립된 상태라 문제는 없었지만, 처음이었다면 위험.

## What Was Surprising

- **SKILL.md가 339줄로 500줄 한도에 여유가 있었다.** Progressive disclosure가 잘 작동해서 모든 Stage의 상세 로직을 references/로 분리한 덕분. 16개 reference 파일이 실제 로직을 담고 SKILL.md는 orchestration overview만.

- **사용자가 Phase 7 이후부터는 "너가 결정"을 선택했다.** Inner Loop의 패턴이 확립되자 사용자가 신뢰를 갖고 재량을 위임. discuss-phase가 짧아지면서 Phase 7-10은 빠르게 진행.

- **단일 세션에 114 commits, 21,000+ 줄이 가능했다.** Context compaction이 여러 번 일어났을 텐데 GSD의 파일 기반 상태 관리 + subagent 패턴 덕분에 맥락 유실 없이 진행. 이것 자체가 dev-lifecycle이 해결하려는 메모리 문제의 부분적 답.

## Key Decisions

ADR은 생성하지 않았지만, 기록할 가치가 있는 주요 결정들:

1. **dev-lifecycle이 spec+prototype을 독립 생성** — GSD/PDCA를 호출하지 않고 직접 생성. GSD는 프로젝트 계획의 주도, dev-lifecycle은 기능 명세의 주도.

2. **E2E Spec + Prototype은 별도 Phase가 아닌 Inner Loop 내장** — PLAN에서 생성, DO에서 체크리스트, TEST에서 검증. 이것이 "100% 기능 실패" 문제의 핵심 해결책.

3. **Deviation 발생 시 사용자 확인 필수** — 자동 진행하지 않고 반드시 사용자에게 알리고 승인 후 계속.

4. **모든 설정 변경 자동 기록** — ADR보다 가벼운 한 줄 기록으로 사각지대 해소.

5. **Living State Document** — 세션 시작 시 한 파일로 전체 맥락 복원. SessionStart hook으로 자동 로드.

## Technical Debt Introduced

- **실제 프로젝트 테스트 미수행** — 모든 산출물이 스킬 정의 문서(Markdown/JSON/HTML)이지 실행 코드가 아님. 실제 프로젝트에서 이 스킬을 invoke했을 때 제대로 작동하는지 아직 미검증.

- **Phase 3 warning: user-login.prototype.html에 e2e-login-005가 두 곳에 중복** — Phase 6 TEST에서 manifest 검증 시 문제될 수 있음. 수정 필요.

- **PDCA skill-invocation.md에 "supplement GSD planning" 스테일 문구** — Phase 4에서 dev-lifecycle이 primary로 변경됐는데 이 문구가 남아있음.

- **SessionStart hook이 `.claude/settings.json`에 프로젝트 로컬로 설정** — 글로벌 설정과의 충돌 가능성 미검증.

## Lessons for Next Phase (v2)

1. **핵심 기능은 처음부터 파이프라인에 내장할 것** — "나중에 추가"로 분리하면 사용자가 다시 지적하게 된다. User Interaction Prototype이 이 교훈.

2. **"먼저 만들고 역검토" 순서를 지킬 것** — 솔루션 설계부터 하면 끼워맞추기가 된다. 실사용 후 검증이 올바른 순서.

3. **discuss-phase에서 사용자가 "너가 결정"을 선택하면 빠르게 진행** — 패턴이 확립된 후에는 재량 위임이 효율적.

4. **Plan checker의 revision loop를 신뢰할 것** — blocker가 나오면 고치고 재검증하는 것이 품질 게이트로 작동한다.

5. **실제 프로젝트 적용이 진짜 테스트** — 스킬 정의 문서 작성은 "설계"일 뿐, "검증"은 실사용에서만 가능.
