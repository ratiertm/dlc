# 프로젝트 회고: dev-lifecycle v2.0 전체

- **Date**: 2026-03-27
- **Phase**: v2.0 마일스톤 (Phase 11-15)
- **Duration**: 2026-03-23 ~ 2026-03-27 (5일, 복수 세션)
- **Commits**: 40
- **Files Changed**: 49
- **Lines Added**: ~6,530

## What Went Well

- **v1.0 교훈이 v2.0에 정확히 적용되었다.** v1.0 회고의 5가지 교훈(핵심 기능 내장, 역검토 순서, 재량 위임, checker 신뢰, 실사용 검증)이 v2.0 작업 방식에 반영됨. 특히 "먼저 만들고 역검토" 패턴이 Phase 14-15에서 리서치→플랜→즉시 실행으로 자연스럽게 녹아들었다.

- **progressive disclosure 패턴이 확장성을 증명했다.** SKILL.md가 339줄(v1.0)에서 383줄(v2.0)로 단 44줄만 증가하면서 5개 대형 기능(config, iteration, upgrade, observability, ecosystem)을 추가. references/ 분리 전략이 v2.0에서도 완벽하게 작동.

- **Phase 14-15를 한 세션에서 연속 실행할 수 있었다.** Phase 14(5개 요구사항, 2개 플랜)와 Phase 15(3개 요구사항, 2개 플랜)를 약 10분 이내에 완료. 패턴이 충분히 확립되어 리서치-플랜-실행 사이클이 기계적으로 진행 가능.

- **기존 인프라 재활용이 효과적이었다.** Phase 15의 `proactive` 설정은 Phase 11에서 이미 lifecycle-config에 정의됨. Phase 14의 JSONL 데이터는 기존 `.lifecycle/history/` 패턴을 보완. 새로 발명하지 않고 기존 구조 위에 쌓는 접근이 빠르고 안전.

## What Went Wrong

- **Phase 14 플랜이 checker feedback으로 한 번 수정되었다.** 초기 플랜에서 일부 인터페이스 정의가 불완전했고, 기존 파일 구조에 대한 가정이 틀렸음. checker가 잡아줬지만, 기존 파일을 먼저 읽고 플래닝했으면 revision이 불필요했을 것.

- **PROJECT.md의 v2.0 체크리스트가 업데이트되지 않았다.** REQUIREMENTS.md와 ROADMAP.md는 완료 상태를 반영했지만 PROJECT.md의 "Active (v2.0)" 항목은 아직 `[ ]` 상태. 메타데이터 동기화 포인트가 3곳(PROJECT.md, REQUIREMENTS.md, ROADMAP.md)으로 분산되어 있어 업데이트 누락이 발생.

- **v2.0 기능의 실사용 검증이 여전히 미수행.** v1.0 회고의 교훈 5번("실사용이 진짜 테스트")이 v2.0에서도 그대로 적용됨. config.yaml 해석, mini-verify 루프, upgrade 마이그레이션, JSONL 분석, 스킬 추천 -- 모두 문서 수준에서 정의됐지만 실제 프로젝트에서 invoke 시 동작 여부는 미확인.

## What Was Surprising

- **SKILL.md 44줄 증가로 5개 기능을 수용한 것.** 500줄 한도 대비 383줄(77%). references/ 파일이 v1.0의 16개에서 21개로 증가했지만 SKILL.md 자체는 거의 변하지 않음. 이 아키텍처가 v3.0 이후에도 유효할 수 있음을 시사.

- **Phase 11-13은 각각 2일씩, Phase 14-15는 한 세션에 완료.** 초반 Phase(config, iteration, upgrade)는 리서치와 checker revision에 시간이 걸렸지만, 후반 Phase(observability, ecosystem)는 패턴이 완전히 확립되어 압축적으로 진행. v1.0에서도 동일한 가속 패턴이 관찰됨.

- **v2.0 요구사항 19개 중 기각된 것이 0개.** 모든 요구사항이 그대로 구현됨. v1.0에서 요구사항을 잘 정의한 덕분에 v2.0에서도 scope creep이 없었음.

## Key Decisions

1. **YAML for config (not JSON/TOML)** -- 사람이 읽기 쉽고 주석을 지원. Phase 11에서 결정.

2. **JSONL for analytics (not JSON array)** -- append-only 안전성. 부분 읽기 가능. Phase 14에서 결정.

3. **mini-verify를 DO 내부 sub-step으로 삽입 (별도 step이 아닌)** -- 기존 워크플로를 최소한으로 변경. Phase 12에서 결정.

4. **Upgrade에 dot-prefixed 마커 파일 사용** -- `.v2-migrated`, `.dlc-version` 등 숨김 파일 컨벤션. Phase 13에서 결정.

5. **Ecosystem 추천은 reference 파일의 테이블 수정만으로 확장** -- SKILL.md나 stage-transitions.md 변경 없이 새 스킬 추가 가능. Phase 15에서 결정.

## Technical Debt Introduced

- **PROJECT.md v2.0 체크리스트 미반영** -- 아직 `[ ]` 상태의 항목들이 남아있음. REQUIREMENTS.md가 source of truth이지만 혼동 가능.

- **References 21개 파일의 상호 참조 일관성 미검증** -- 파일이 늘어나면서 cross-reference 오류 가능성 증가. Architecture Audit 주석은 SKILL.md만 커버.

- **v1.0 회고에서 지적된 tech debt 3건이 아직 미해결** -- user-login.prototype.html의 중복 ID, PDCA skill-invocation.md 스테일 문구, SessionStart hook 충돌 가능성.

- **실사용 검증 미수행 (v1.0과 동일)** -- 스킬 정의 문서의 완성도는 높으나, 실제 프로젝트에서 전체 lifecycle을 돌려본 적 없음.

## Lessons for Next Phase (v3)

1. **메타데이터 동기화 포인트를 줄여야 한다.** PROJECT.md, REQUIREMENTS.md, ROADMAP.md 세 곳에 상태를 유지하면 불일치가 발생. single source of truth 원칙을 강화하거나, 자동 동기화 로직이 필요.

2. **실사용 검증을 v3.0 첫 Phase로 설정해야 한다.** 두 번의 마일스톤 회고에서 모두 "실사용 미검증"이 지적됨. v3.0은 실제 프로젝트에 적용하여 검증하는 것으로 시작해야 함.

3. **v1.0 tech debt를 정리하는 하우스키핑 Phase가 필요하다.** 3건의 미해결 tech debt가 누적 중. 새 기능 전에 정리.

4. **References 파일 간 cross-reference 검증 자동화를 고려할 것.** 21개 파일이 서로를 참조하면서 수동 일관성 유지가 어려워지고 있음.

5. **Phase 가속 패턴을 신뢰할 것.** v1.0과 v2.0 모두 후반 Phase에서 급격한 가속이 발생. 초반에 패턴을 잡는 데 시간이 걸리는 것은 정상이며, 후반 가속으로 보상됨.
